# コーディング規約・命名規則

対象：AIブライダルプランニング支援システム（Creepy Code）
対象範囲：バックエンド（Java / Spring Boot）・フロントエンド（Thymeleaf / HTML / CSS）の両方

---

## 1. 命名規則

### バックエンド（Java）

| 対象 | ルール | 例 |
|------|--------|-----|
| クラス名 | パスカルケース | `CustomerService`、`PlanController` |
| メソッド名 | キャメルケース、動詞始まり | `getCustomerById`、`calculatePrice` |
| 変数名 | キャメルケース | `customerName`、`totalPrice` |
| 定数 | 全て大文字＋アンダースコア | `MAX_PLAN_COUNT` |
| パッケージ名 | 全て小文字 | `com.creepycode.service` |
| Entityクラス | テーブル名を単数形パスカルケース | `Customer`（テーブル`customers`に対応） |
| Repositoryインターフェース | `〇〇Repository` | `CustomerRepository` |
| Serviceクラス | `〇〇Service` | `CustomerService` |
| Controllerクラス | `〇〇Controller` | `CustomerController` |

### フロントエンド（Thymeleaf / HTML / CSS）

| 対象 | ルール | 例 |
|------|--------|-----|
| HTMLファイル名 | ハイフン区切り小文字 | `plan-simulation.html` |
| idの命名 | キャメルケース | `id="totalPrice"` |
| classの命名 | ハイフン区切り小文字（ブロック単位） | `class="plan-card__title"` |
| Thymeleafの変数名 | キャメルケース | `th:text="${customerName}"` |

## 2. パッケージ・ディレクトリ構成

### バックエンド

```
com.creepycode
 ├─ controller   … 画面・APIの入り口
 ├─ service      … 業務ロジック
 ├─ repository   … DBアクセス
 ├─ entity       … テーブルに対応するクラス
 ├─ dto          … 画面⇄サーバー間のデータの入れ物
 └─ config       … 設定関連
```

実装する順番は、これまでのチーム開発ルールと同じく
`テーブル確認 → Entity → Repository → Service → Controller → 画面 → 動作確認` とする。

### フロントエンド（テンプレート）

```
src/main/resources/templates
 ├─ layout      … 共通ヘッダー・フッターなどの土台
 ├─ customer    … 顧客向け画面
 ├─ planner     … プランナー向け画面
 └─ fragments   … 使い回すパーツ（ボタン・カードなど）
```

---

## 3. フォーマット

- インデントは半角スペース4つ（バックエンド・フロントエンド共通）
- 1行は120文字を目安に収める
- 中括弧`{`は行の末尾に置く（改行しない）
- CSSは1セレクタにつき1プロパティ1行

---

## 4. コメントの書き方

- publicメソッドには「何をするメソッドか」を1行で書く
- 複雑な処理には理由（なぜその実装にしたか）をコメントする
- TODOコメントは `// TODO: 内容（担当者名）` の形式で残す

---

## 5. Thymeleaf特有のルール

- 共通レイアウトは `layout:decorate` で必ず継承する（同じヘッダー・フッターを毎回書かない）
- 繰り返し使うパーツ（料金カードなど）は `fragments` に切り出す
- 画面側に複雑な判定ロジックを書かない（複雑な処理はController・Serviceで済ませ、HTML側は`th:if`程度に留める）

---

## 6. 例外処理・ログ

- 想定される異常系は独自の例外クラスを作る（例：`CustomerNotFoundException`）
- ログは `System.out.println` を使わず、SLF4Jのロガーを使う

---

## 7. その他

- 色・フォントサイズなどはCSS変数にまとめ、あちこちに散らばらせない
- id指定でのスタイル付けは避け、class指定を基本にする

---

## 既存ルールとの整合性について

この規約は、以下の既存ルールと矛盾しないことを確認済みです。

- ブランチ命名（`feature/〇〇`・`fix/〇〇`・`docs/〇〇`）
- コミットメッセージ形式（`feat:`・`fix:`・`docs:`など）
- Springの実装順序（テーブル確認 → Entity → Repository → Service → Controller → 画面 → 動作確認）
